# 需求说明

## 1. 产品定位

POP（Proxy of Proxy）是一个面向个人使用场景的本地 HTTP 代理服务，并提供配套 Web Console 用于配置和运行期观测。

核心目标按优先级排序：

1. 易用性：用户可直接将系统代理指向 POP，并通过 Console 完成主要配置。
2. 可维护性：规则、上游、代理、持久化、观测、控制台职责边界清晰。
3. 资源友好：长期运行时内存、连接和活动数据规模可控。
4. 性能满足个人流量场景，不以高并发吞吐为目标。

## 2. 当前产品范围

### 2.1 已实现范围

- 本地 HTTP 代理服务，支持普通 HTTP 请求和 `CONNECT` 隧道。
- 基于域名规则的路由决策：`DIRECT` / `PROXY` / `BLOCK`。
- 多上游 HTTP 代理配置与按规则分流。
- SQLite 持久化保存规则与上游配置。
- Web Console 页面与 Console API。
- 运行期 telemetry：统计、活动列表、SSE 实时流。
- PAC 文件生成：`GET /proxy.pac`。
- 数据管理：备份、恢复、ABP 文本导入。

### 2.2 当前不在范围内

- SOCKS5 上游代理。
- Web Console 登录认证与权限控制。
- Console UI 自动化测试。
- 历史统计持久化、报表系统。
- 多租户与复杂 RBAC。

## 3. 典型使用场景

1. 用户将操作系统或浏览器代理设置为 POP 的代理地址。
2. 访问内网域名时，POP 直接连接目标站点。
3. 访问外网域名时，POP 按规则转发到不同上游 HTTP 代理。
4. 对广告或不希望访问的域名，POP 直接本地阻断并返回错误码。
5. 未命中规则的流量按默认动作处理，当前默认值为 `DIRECT`。
6. 用户通过 Web Console 查看实时活动、基础统计，并管理规则与上游。
7. 用户可导出 SQLite 数据备份、恢复数据，或批量导入 ABP 域名规则。
8. 需要自动配置代理时，可通过 `GET /proxy.pac` 获取 PAC 文件。

## 4. 功能需求

### 4.1 代理能力

- 提供 HTTP 代理监听端口。
- 支持普通 HTTP 请求代理。
- 支持 `CONNECT` 隧道代理。
- 对 `DIRECT` 和 `PROXY` 两类路径都应能处理普通 HTTP 与 `CONNECT`。

### 4.2 路由决策

- 路由输入基于归一化后的目标 host。
- 规则匹配语义为“匹配根域及其所有子域名”。
- 支持 legacy `*.example.com` 兼容写法，但内部语义等同于 `example.com`。
- 多条规则命中时，按最长 pattern 优先。
- 若 pattern 长度相同，则按当前存储顺序决策；当前实现等价于较新的规则优先。
- 未命中任何规则时，使用默认动作。
- 默认动作当前默认为 `DIRECT`，同时配置模型允许 `DIRECT` / `PROXY` / `BLOCK`。

### 4.3 规则模型

规则字段包括：

- `id`：数据库主键。
- `enabled`：是否启用。
- `pattern`：域名模式。
- `action`：`DIRECT` / `PROXY` / `BLOCK`。
- `upstream_id`：`PROXY` 动作时关联上游主键。
- `block_status`：`BLOCK` 动作时使用的状态码。
- `created_at`：创建或最近一次按同 pattern 覆盖更新的时间。

规则行为要求：

- 规则支持新增、修改、删除、启用和禁用。
- 新建规则时，如存在相同 pattern 的规则，应覆盖原规则并刷新 `created_at`，而不是新增重复规则。
- Web Console 创建或编辑 `BLOCK` 规则时，状态码固定为 `404`。
- 规则列表接口支持分页与按 `pattern` 关键字搜索。
- 规则重排接口路径已保留，但当前版本不要求实际生效；接口返回 `reorder_disabled`。

### 4.4 上游代理

- 支持配置多个上游代理。
- 上游字段包括 `id`、可选 `name`、`url`、`enabled`。
- 当前版本仅支持 `http://` 上游代理。
- 当规则引用不可用或不存在的上游时，请求返回 `502 Bad Gateway`。
- 已被规则引用的上游不允许删除。

### 4.5 配置与运行参数

- 配置来源为命令行参数、环境变量和内建默认值。
- 配置优先级固定为 `CLI > ENV > default`。
- 当前运行配置至少包括：
  - `proxy_listen`
  - `console_listen`
  - `sqlite_path`
  - `default_action`
  - `pac_proxy_addr`（可选 PAC 输出代理地址覆盖）
- 配置更新后，应校验并重建 matcher 与 upstream manager，再应用到运行态。

### 4.6 持久化与数据管理

- 规则与上游配置持久化到 SQLite。
- 服务重启后应从 SQLite 恢复规则与上游配置。
- 统计与活动数据属于运行态数据，不需要持久化。
- 系统应支持全量数据备份与恢复。
- 备份数据需带 `data_format_version`，恢复时需校验兼容性。
- 数据恢复采用全量替换策略。
- 支持导入 ABP 文本规则，当前导入目标使用统一 `route_target`：
  - `DIRECT`
  - 指定 `UPSTREAM`

### 4.7 Web Console 与 Console API

- 当前版本 Web Console 无认证。
- Console 提供页面：
  - Stats
  - Activities
  - Rules
  - Upstreams
  - Data
- Console API 覆盖以下能力：
  - 配置读取与更新
  - 上游 CRUD
  - 规则 CRUD
  - 规则分页搜索
  - 数据备份与恢复
  - ABP 导入
  - 版本信息读取
  - stats / activities / SSE
- 活动页支持基于活动快速补充规则。

### 4.8 PAC 能力

- 提供 `GET /proxy.pac` 返回 PAC 文件。
- PAC 内容应基于当前规则和默认动作动态生成。
- `DIRECT` 规则生成 `DIRECT`。
- `PROXY` 规则生成指向 POP 代理地址的 `PROXY host:port`。
- `BLOCK` 规则在 PAC 中使用不可达代理地址表达阻断效果。
- 如配置了 `pac_proxy_addr`，PAC 输出中应优先使用该地址，而不是从请求地址推导。

### 4.9 运行期观测

- 提供运行统计快照接口。
- 提供最近活动列表接口。
- 提供活动 SSE 实时流接口。
- 统计至少覆盖：
  - in-flight
  - 总请求数
  - 错误数
  - 入/出字节
- 活动事件至少包含客户端、方法、目标 host、动作、规则 ID、状态码、耗时、字节数等信息。

## 5. 非功能需求

### 5.1 可维护性

- 模块职责清晰，至少包括配置、规则、代理、上游、持久化、观测、控制台。
- 关键行为必须具备自动化测试覆盖。
- PAC 生成与 HTTP 代理兼容性必须纳入常规测试范围。

### 5.2 资源约束

- 活动数据必须使用有界内存模型。
- 活动存储需满足 ring buffer 容量上限与 TTL 清理。
- 统计只保留聚合值，不保存无限增长历史。
- 连接应复用，并设置合理超时。

### 5.3 安全基线（当前版本）

- Web Console 当前不做认证。
- Console 默认监听地址应保持本机回环地址，降低暴露风险。
- 不实现复杂用户系统、细粒度权限和审计要求。

## 6. 验收标准

- 将系统代理指向 POP 后，可正常转发 HTTP 和 `CONNECT` 请求。
- 内部域名可按规则走 `DIRECT`。
- 外部域名可按规则分别走不同上游 HTTP 代理。
- 广告或阻断域名按规则返回阻断状态码；Web Console 创建的阻断规则固定为 `404`。
- 未命中规则时按默认动作执行，当前默认值为 `DIRECT`。
- Console API 可完成配置读取更新、规则与上游 CRUD、规则搜索分页、数据备份恢复、ABP 导入。
- 可通过 `GET /proxy.pac` 获取与当前规则一致的 PAC 文件。
- 可查看实时活动、基础统计和 SSE 活动流。
- 服务重启后，SQLite 中的规则和上游配置可被正确恢复。
